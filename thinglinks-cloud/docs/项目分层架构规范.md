# 系统架构概览

分层架构模式（Layered Architecture Pattern）

## 架构分层

- Controller层 (表现层) → 输入：VO | 输出：ResultVO
- Service层 (业务层) → 输入：DTO | 输出：ResultDTO
- Manager层 (持久层) → 输入：DO | 输出：DO、ResultDO

说明：定制化复杂业务部分建议采用上述方式进行定义，如果是简单的业务场景单表的CRUD，
Controller 可直接透传VO，可以省略Manager层，直接在Service层进行数据库操作，并返回ResultVO。

## 各层职责说明

### Manager层 (持久层)

**核心职责：**

- 负责与数据库的直接交互
- 封装所有数据访问逻辑
- 处理基础的数据增删改查操作

**数据对象规范：**

- **基础实体**：XxxDO (Data Object)
    - 与数据库表结构完全对应
    - 包含所有数据库字段
- **查询结果**：XxxResultDO (Result Data Object)
    - 包含关联查询结果
    - 可能包含计算字段

### Service层 (业务层)

**核心职责：**

- 处理核心业务逻辑
- 协调多个Manager完成复杂业务操作
- 事务管理
- 业务规则验证

**数据对象规范：**

- **输入参数**：XxxDTO (Data Transfer Object)
    - Service方法的输入参数
    - 包含业务操作所需的所有字段
- **返回结果**：XxxResultDTO (Result Data Transfer Object)
    - Service方法的返回结果
    - 包含业务处理结果和状态信息

### Controller层 (表现层)

**核心职责：**

- 接收HTTP请求
- 参数校验和格式化
- 调用Service层处理业务
- 返回统一格式的响应

**数据对象规范：**

- **请求参数**：XxxVO (Request View Object)
    - 接收前端请求参数
    - 包含参数校验注解
- **返回结果**：XxxResultVO (Response View Object)
    - 返回给前端的数据
    - 包含前端展示需要的格式化字段

## 数据流转流程

1. **请求进入**：前端发送请求，Controller接收VO
2. **参数转换**：Controller将VO通过Convert转换为DTO，调用Service层
3. **业务处理**：Service接收DTO，执行业务逻辑，调用Manager层
4. **数据访问**：Manager接收DO参数，操作数据库，返回ResultDO
5. **结果返回**：Service将ResultDO通过Convert转换为ResultDTO，返回给Controller
6. **响应输出**：Controller将ResultDTO通过Convert转换为ResultVO，返回给前端

## 命名规范总结

| 层级         | 输入对象 | 输出对象      | 说明           |
|------------|------|-----------|--------------|
| Controller | VO   | ResultVO  | 视图对象，面向前端    |
| Service    | DTO  | ResultDTO | 数据传输对象，业务层使用 |
| Manager    | DO   | ResultDO  | 数据对象，持久层使用   |
