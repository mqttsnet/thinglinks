# 第一部分：架构规范

## 1. 技术栈

| 类别 | 技术 |
|------|------|
| JDK | **17**（强制使用 JDK 17 特性） |
| 框架 | Spring Boot 3.x + Spring Cloud |
| 注册/配置中心 | Nacos |
| 网关 | Spring Cloud Gateway（WebFlux） |
| RPC | OpenFeign（微服务） / 本地调用（boot-impl） |
| ORM | MyBatis-Plus |
| 多租户数据源 | dynamic-datasource + `@DS` |
| 分布式 ID | 雪花算法 |
| 缓存 | Redis + CachePlusOps |
| 分布式锁 | DistributedLock |
| 任务调度 | XXL-Job（iot-executor） |
| 线程池 | DynamicTp |
| 参数校验 | ArgumentAssert + BizException |
| 文件存储 | FileFacade（Feign） |

## 2. 模块分层结构

每个业务模块（link、video、base 等）统一采用以下结构：

```
thinglinks-xxx/
├── thinglinks-xxx-entity/          # 实体层：VO、Entity、DTO、枚举、CacheVO
├── thinglinks-xxx-biz/             # 核心业务层：Service、Manager、Mapper
├── thinglinks-xxx-biz-protocol/    # 协议层（可选）：进程绑定的协议栈
├── thinglinks-xxx-controller/      # 控制器层：REST 接口
├── thinglinks-xxx-facade/          # Facade 层
│   ├── thinglinks-xxx-api/         #   接口定义
│   ├── thinglinks-xxx-boot-impl/   #   单体版实现（executor 本地调用）
│   └── thinglinks-xxx-cloud-impl/  #   微服务版实现（Feign 远程调用）
└── thinglinks-xxx-server/          # 启动模块：SpringBoot Application
```

---

## 3. 分层调用链路

```
Controller / Listener / Commander / Job
        ↓
    ServiceImpl（@DS 多租户数据源路由）
        ↓
    ManagerImpl（数据访问封装）
        ↓
      Mapper（MyBatis）
```

**强制约束**：
- 所有数据库操作必须经过 `ServiceImpl` 层（标注 `@DS(DsConstant.BASE_TENANT)`）
- **禁止**跨层直接调用 Manager 或 Mapper
- ManagerImpl **不标注** `@DS`（全模块统一）

数据流转详见 [项目分层架构规范](项目分层架构规范.md)。

## 4. Facade 双部署模式

每个业务模块提供两种 Facade 实现，支持单体和微服务双模式：

| 模式 | 实现模块 | 调用方式 | 使用场景 |
|------|---------|---------|---------|
| boot-impl | `xxx-boot-impl` | 直接注入 Service，本地方法调用 | iot-executor 本地调用 |
| cloud-impl | `xxx-cloud-impl` | 注入 Feign Client，远程 HTTP 调用 | 微服务间调用 |

### 接口定义（api）

```java
public interface VideoJobHandlerFacade {
    R<?> zlmMediaServerHeartbeat(Long tenantId);
}
```

### 本地调用实现（boot-impl）

```java
@Service
@RequiredArgsConstructor
public class VideoJobHandlerFacadeImpl implements VideoJobHandlerFacade {
    private final ZLMMediaServerStatusManager statusManager;

    @Override
    public R<?> zlmMediaServerHeartbeat(Long tenantId) {
        statusManager.executeForTenant(tenantId);
        return R.success();
    }
}
```

### 远程调用实现（cloud-impl）

```java
@Service
public class VideoJobHandlerFacadeImpl implements VideoJobHandlerFacade {
    @Lazy @Autowired
    private VideoJobHandlerApi videoJobHandlerApi;

    @Override
    public R<?> zlmMediaServerHeartbeat(Long tenantId) {
        return videoJobHandlerApi.zlmMediaServerHeartbeat(tenantId);
    }
}
```

### 依赖引入规则

- 微服务 A 调用微服务 B 的接口 → 在 `a-server/pom.xml` 加入 `b-cloud-impl` 依赖
- cloud-impl 中的 Feign Client **必须 `@Lazy` 延迟注入**

## 5. 协议层隔离

### 模块划分标准

| 判断条件 | 归属模块 |
|---------|---------|
| 需要 `SIPSender`、`Commander`、`SipLayer` 等协议 Bean | `xxx-biz-protocol` |
| 纯业务逻辑（CRUD、缓存管理、流媒体节点管理） | `xxx-biz` |
| 端口监听、协议栈、SIP 指令收发 | `xxx-biz-protocol` |

### executor 安全约束

`iot-executor` 只依赖 `xxx-boot-impl` → `xxx-biz`，**绝不能引入 `xxx-biz-protocol`**。

原因：协议模块中的 `CommandLineRunner`（如 `SipLayer`）会在启动时监听端口，executor 中无相关配置会导致 NPE。通过 Maven 模块隔离，协议类天然不在 executor classpath 中，无需 `@ComponentScan` 排除。

### 新增协议接入

1. 在 `biz-protocol` 中创建协议包（如 `onvif/`）
2. 实现 `DeviceAccessProtocol` 接口
3. 在 `protocol/` 包中注册协议适配器
4. 协议相关的事件（定义、发布、监听）全部放在协议包内，**严禁渗透到 `biz`**
5. 无需修改 executor 或其他模块

## 6. 事件驱动架构

### 事件定义层次

| 层级 | 事件类型 | 示例 | 说明 |
|------|---------|------|------|
| entity | 通用领域事件 | `MediaServerOnlineEvent`、`DeviceInfoOnlineEvent` | 协议无关，biz 和 protocol 共用 |
| biz | 流媒体 Hook 事件 | `MediaArrivalEvent`、`MediaRecordMp4Event` | ZLM/ABL Hook 回调触发 |
| biz-protocol | 协议专属事件 | `SipCommandSentEvent`、`InviteSessionCreatedEvent` | GB28181/ISUP/JT1078 协议栈内部 |

### 核心规则

- **状态变更优先使用事件驱动**，不直接操作 Redis 或跨层调用
- **协议事件严禁往下渗透**：SIP/GB28181/ISUP/JT1078 相关事件必须在 `biz-protocol` 内闭环
- **biz 层事件必须协议无关**：`@EventListener` 不得依赖协议 Bean
- **跨层通信通过 entity 事件**：协议层通知业务层时，使用 entity 层定义的通用事件

### executor 中的事件安全性

biz 层的事件监听器会被加载到 executor 进程中，安全性保证：

| 监听器 | executor 中行为 | 安全性 |
|--------|---------------|--------|
| `MediaServerStatusEventListener` | StatusManager 触发，做 DB 状态更新，有 `@DS` 路由 | ✅ 安全 |
| `HookZlmServerKeepaliveEventListener` | 无 ZLM Hook 回调源 | ✅ 不会触发 |
| `MediaRecordMp4EventListener` | 无录像 Hook 回调源 | ✅ 不会触发 |
| `HookSubscribe` | 内存订阅管理，无订阅注册 | ✅ 空转无副作用 |

## 7. 定时任务归属

### 迁移判断标准

| 任务类型 | 归属 | 调度方式 |
|---------|------|---------|
| 纯业务逻辑（CRUD、缓存刷新、心跳检测） | `iot-executor` | XXL-Job |
| 进程绑定的协议逻辑（SIP 消息队列、Hook 超时清理） | 各业务 server | `@Scheduled` |

### XXL-Job Handler 实现

```java
@Component
@RequiredArgsConstructor
public class VideoJob {
    private final VideoJobHandlerFacade videoJobHandlerFacade;

    @XxlJob("zlmMediaServerHeartbeatJobHandler")
    public void zlmMediaServerHeartbeatJobHandler() {
        videoJobHandlerFacade.zlmMediaServerHeartbeat(tenantId);
    }
}
```

### 保留在 server 本地的 @Scheduled（不迁移）

| 场景 | 频率 | 保留原因 |
|------|------|---------|
| SIP 协议实时消息队列 | 200ms | 进程内消息消费 |
| Hook 订阅清理 | 5min | 绑定 Hook 接收器的内存数据 |
| 内存任务生命周期管理 | 5min | DynamicTask 进程内 |
| SipStack Redis 续期 | 60s | SipStack 进程绑定 |

## 14. 枚举与字典回显

### @Echo 注解

在 `ResultVO` 中对字典字段添加 `@Echo`，由 `echoService.action(resultVO)` 自动回显：

```java
public class VideoRecordPlanResultVO {

    @Echo(api = EchoApi.DICTIONARY_ITEM_FEIGN_CLASS,
          dictType = EchoDictType.Video.VIDEO_RECORD_PLAN_TYPE)
    @Schema(description = "计划类型")
    private Integer planType;

    @Echo(api = EchoApi.DICTIONARY_ITEM_FEIGN_CLASS,
          dictType = EchoDictType.Video.VIDEO_RECORD_FILE_FORMAT)
    @Schema(description = "录制格式")
    private String recordFormat;
}
```

### 前端配合

- **列表展示**：`dataIndex` 使用 `'echoMap.xxx'`
- **编辑表单**：组件使用 `ApiSelect` + `dictComponentProps(DictEnum.XXX)`

### 禁止手动拼 desc

```java
// ❌ 禁止
resultVO.setPlanTypeDesc(PlanTypeEnum.getDesc(entity.getPlanType()));

// ✅ 正确：依赖 @Echo 自动回显
echoService.action(resultVO);
```

## 15. 多租户数据源

### @DS 注解规则

```java
@Service
@DS(DsConstant.BASE_TENANT)  // "#thread." + TENANT_BASE_POOL_NAME_HEADER
public class VideoMediaServerServiceImpl implements VideoMediaServerService {
```

| 类型 | 是否标注 @DS | 说明 |
|------|:-----------:|------|
| ServiceImpl | ✅ | 数据源切换的**唯一入口** |
| ManagerImpl | ❌ | 全模块统一不标注 |
| StatusManager（Job 入口 Bean） | ✅ | 确保 ThreadLocal 数据源在调用 ServiceImpl 前已设置 |

### DsConstant 定义

```java
BASE_TENANT = "#thread." + TENANT_BASE_POOL_NAME_HEADER
```

从 ThreadLocal 读取当前租户的数据源名称，实现多租户路由。

## 17. 缓存

### CacheKey 构建器

每个模块的缓存 Key 通过 `CacheKeyBuilder` 统一管理：

```java
public class SipServerInfoCacheKeyBuilder implements CacheKeyBuilder {
    @Override
    public String getPrefix() {
        return "video:sip:server:";
    }
}
```

### 缓存对象命名

CacheKey 存储的实体统一使用 `XxxCacheVO` 命名，放在 `entity/cache/` 包下。

### 操作方式

通过 `CachePlusOps` 操作缓存，**不直接使用 `RedisTemplate`**：

```java
CacheKey cacheKey = SipServerInfoCacheKeyBuilder.build(instanceId);
cachePlusOps.set(cacheKey, sipServerInfo);
```

## 18. 分布式锁

使用框架内置的 `DistributedLock.tryLockAndRun()` 模式：

```java
// LockKeyBuilder 统一定义锁
public class VideoLockKeyBuilder {
    public static CacheKey forRecordPlanExecute(Long planId) { ... }
}

// 使用
CacheKey lockKey = VideoLockKeyBuilder.forRecordPlanExecute(planId);
LockRunResult<?> result = distributedLock.tryLockAndRun(
    lockKey.getKey(),
    lockKey.getExpire().getSeconds(),
    TimeUnit.SECONDS,
    () -> doExecutePlan(plan)
);
if (!result.isLocked()) {
    log.info("录像计划正在被其他实例执行: planId={}", planId);
}
```

## 19. traceId 链路追踪

### 自动传递

请求经过 Gateway + `HeaderThreadLocalInterceptor` 后自动生成 traceId。内部 Feign 调用通过 `FeignAddHeaderRequestInterceptor` 自动透传。

### 手动生成

非网关入口的场景（如 Job 执行、后台线程），手动设置：

```java
ContextUtil.setLogTraceId(IdUtil.fastSimpleUUID());
```

# 第四部分：网关与权限规范

## 21. Gateway 网关约束

### 依赖约束

| 规则 | 说明 |
|------|------|
| ❌ 不能有 `thinglinks-validator-starter` | 与 Gateway 冲突 |
| ❌ 不能有 `spring-webmvc` | Gateway 基于 WebFlux |
| ❌ 不能使用构造器注入 | Gateway 限制 |
| ✅ 必须有 `thinglinks-log-start` | 日志链路追踪 |
| ✅ Feign 必须 `@Lazy` 延迟注入 | 避免循环依赖 |

### 注意事项

- `thinglinks-base-server` 不能加入 `spring-boot-starter-tomcat` 依赖，否则 WebSocket 冲突
- 添加租户时，需要初始化网关路由

---

## 22. 接口权限分类

| 权限类型 | 路径模式 | 需要 Tenant ID | 需要登录 | 验证 URI 权限 | 典型场景 |
|---------|---------|:-:|:-:|:-:|------|
| `anyone` | `/*/anyone/**` | ✅ | ✅ | ❌ | 用户个人中心 |
| `anyUser` | `/*/anyUser/**` | ✅ | ❌ | ❌ | 公共信息查询 |
| `anyTenant` | `/*/anyTenant/**` | ❌ | ❌ | ❌ | Hook 回调、全局配置 |
| `inner` | `/*/inner/**` | 按需透传 | ❌ | ❌ | 服务间 Feign 内部 RPC |

- Hook 回调接口统一走 `anyTenant` 权限路径
- 内部服务调用统一走 `inner` 权限路径,由 Feign / Nacos 直连;外部请求经网关命中 `/inner/**` 会被拒绝
- 详见 [接口权限配置规范](接口权限配置规范.md)
